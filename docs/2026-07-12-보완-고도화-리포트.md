# startup-law-mcp 보완·고도화 리포트

- 작성일: 2026-07-12
- 대상 버전: v1.1 (`305de21` 기준)
- 목적: Claude Code에 전달할 구현 작업 명세
- 원칙: 법령·지원사업 데이터의 **정확성, 보존성, 추적 가능성**을 기능 추가보다 우선한다.

## 1. 현재 상태 요약

현재 프로젝트는 로컬 MCP MVP로서 핵심 기능이 동작한다.

- 법령 50개, 조문 8,191개 인덱싱
- K-Startup 공고 310건, 사업소개 511건 보유
- 법령 검색, 조문 조회, 인용 검증, 참조 그래프 CLI 실행 확인
- 법령 매니페스트 오류 및 누락 파일 없음
- 지원사업 이름 누락 및 중복 ID 없음

다만 운영 안정성 관점에서는 다음 위험이 있다.

1. 정상 형태가 아닌 빈 API 응답도 성공으로 간주해 기존 스냅샷을 0건으로 덮어쓸 수 있다.
2. 법령 동기화 실패·삭제·이름 변경 시 오래된 Markdown이 인덱스에 남을 수 있다.
3. MCP 입력값 검증이 없어 빈 검색어와 음수 `limit`이 예상 밖의 대량 결과를 만든다.
4. 사업소개 URL 511건이 `https://` 없는 문자열로 저장되어 직접 링크로 쓰기 어렵다.
5. 일부 인용 검증 로직이 본법과 시행령·시행규칙을 섞어 계산한다.
6. 데이터 스키마, 의존성, 품질 검사와 운영 상태 관찰 기능이 부족하다.

## 3. P0: 지원사업 스냅샷 보호

### 문제

`program_sync.sync()`는 두 API 호출이 예외 없이 빈 배열을 반환하면 이를 정상 결과로 저장한다. API가 HTTP 200 오류 엔벨로프를 반환하거나 필드 구조가 바뀌면 기존 공고·사업소개 파일이 모두 0건으로 교체될 수 있다.

현재 파일 저장도 목적 파일에 직접 `write_text()`를 호출하므로 프로세스 종료, 디스크 오류, 두 번째 파일 저장 실패가 발생하면 두 스냅샷이 서로 다른 상태가 될 수 있다.

### 수정 대상

- `program_sync.py`
- `programs.py`
- `tests/test_program_sync.py`
- `tests/test_programs.py`

### 구현 요구사항

1. API 응답 경계에서 다음을 검증한다.
   - 최상위 응답이 객체인지
   - `data`가 배열인지
   - `totalCount`가 0 이상의 정수로 해석 가능한지
   - 아이템에 종류별 필수 식별자와 이름 필드가 있는지
2. 다음 조건에서는 동기화를 실패로 처리하고 기존 스냅샷을 유지한다.
   - 공고와 사업소개가 모두 0건
   - 필수 필드 누락률이 허용 기준을 초과함
   - 기존 건수 대비 급격한 감소가 발생함
3. 급감 기준은 코드에 흩어진 숫자가 아니라 명명된 정책값으로 둔다. 초기 권장값은 기존 건수 대비 70% 이상 감소다.
4. 신규 결과를 임시 디렉터리에 먼저 기록하고 JSON 재파싱까지 완료한 후 목적 파일로 교체한다.
5. 두 파일에 동일한 `snapshot_id`와 `fetched_at`을 기록한다.
6. 한 파일만 존재하거나 두 파일의 `snapshot_id`가 다르면 조회 결과에 데이터 이상 경고를 포함한다.
7. 동기화 성공 결과에는 이전·신규 건수와 변화량을 반환한다.

권장 결과 예시:

```json
{
  "status": "ok",
  "snapshot_id": "2026-07-12T07:00:00Z",
  "announcements": {"before": 310, "after": 325, "delta": 15},
  "intros": {"before": 511, "after": 512, "delta": 1},
  "fetched_at": "2026-07-12T07:00:00+00:00"
}
```

### 필수 테스트

- API가 빈 `data`를 반환하면 기존 파일 내용이 바뀌지 않는다.
- HTTP 200 + 오류 JSON/XML 응답은 성공으로 처리되지 않는다.
- `data`가 객체 또는 문자열이면 명시적 스키마 오류가 발생한다.
- 첫 번째 임시 파일 저장 후 두 번째 저장이 실패해도 기존 두 파일이 유지된다.
- 두 파일에 동일한 `snapshot_id`가 기록된다.
- 급감 기준을 넘으면 저장을 중단하고 원인을 결과에 포함한다.
- `count`와 실제 `items` 길이가 다르면 조회 시 경고한다.

## 4. P0: 법령 동기화·인덱스 세대 일치

### 문제

`law_sync.sync()`는 동기화에 실패했거나 큐레이션에서 제거된 과거 Markdown 파일을 정리하지 않는다. `law_search.build_index()`는 매니페스트가 아니라 `data/laws/*.md` 전체를 읽기 때문에 오래된 법령이 계속 검색될 수 있다.

또한 `law_sync.py sync`와 `law_search.py build`가 별도 명령이어서 새 법령 파일과 과거 인덱스가 공존할 수 있다.

### 수정 대상

- `law_sync.py`
- `law_search.py`
- `data/sources.json` 스키마
- `tests/test_sync.py`
- `tests/test_parse.py`

### 구현 요구사항

1. 법령 동기화는 staging 디렉터리에서 수행한다.
2. 매니페스트에 포함된 파일만 인덱스 입력으로 사용한다.
3. 전체 동기화가 성공한 경우에만 법령 파일, 매니페스트, 인덱스를 같은 세대로 교체한다.
4. 부분 실패가 허용되는 정책을 유지한다면 다음을 반드시 기록한다.
   - 실패 법령
   - 실패 단계
   - 이전 스냅샷 사용 여부
   - 해당 법령의 마지막 정상 갱신 시각
5. 매니페스트에 다음 필드를 추가한다.
   - `snapshot_id`
   - `fetched_at`
   - `index_built_at`
   - 파일별 `sha256`
   - 전체 동기화 성공 여부
6. 큐레이션에서 제거된 법령은 다음 성공 세대에서 인덱스와 데이터 디렉터리 모두에서 제외한다.
7. 서버 시작 또는 첫 조회 시 매니페스트와 인덱스의 `snapshot_id`가 다르면 검색을 계속하지 말고 명확한 재빌드 오류를 반환한다.
8. 장기 실행 프로세스에서 인덱스를 다시 빌드했다면 메모리 캐시도 같은 시점에 무효화한다.

### 필수 테스트

- 큐레이션에서 법령을 제거한 뒤 동기화하면 검색 결과에서도 사라진다.
- 특정 법령 갱신 실패 시 이전본 사용 여부가 매니페스트에 명시된다.
- 매니페스트에 없는 Markdown은 인덱싱되지 않는다.
- 인덱스와 매니페스트 세대가 다르면 조회가 실패한다.
- 인덱스 재빌드 후 기존 프로세스의 캐시가 최신 결과를 반환한다.
- staging 단계 실패 시 기존 법령 파일과 인덱스가 유지된다.

## 5. P1: MCP 입력값과 반환 계약 검증

### 확인된 재현 사례

- `get_program("")`이 임의의 첫 5건을 반환한다.
- 법령 검색에서 `limit=-1`을 주면 Python 음수 슬라이스 때문에 대량 결과가 반환된다.
- 잘못된 `status`와 `law_type`은 입력 오류 대신 빈 결과처럼 보인다.

### 수정 대상

- `server.py`
- `law_search.py`
- `programs.py`
- 관련 테스트 전체

### 구현 요구사항

1. 모든 문자열 입력은 NFC 정규화와 `strip()`을 경계에서 한 번만 수행한다.
2. `query`, `name`, `source`, `article`의 필수 여부와 최소 길이를 명시한다.
3. `limit`은 정수 1~50으로 제한한다.
4. `status`는 `open`, `closing_soon`, `upcoming`, `closed`만 허용한다.
5. `law_type`은 현재 인덱스에서 지원하는 법종 값으로 제한한다.
6. 잘못된 입력은 빈 배열과 구분되는 구조화된 오류로 반환한다.
7. 내부 함수도 MCP 외부에서 호출될 수 있으므로 핵심 경계에는 동일한 불변조건을 유지한다.
8. 반환형은 가능한 범위에서 `TypedDict`, frozen dataclass 또는 명시적 결과 타입으로 표현한다.

권장 오류 예시:

```json
{
  "status": "invalid_input",
  "field": "limit",
  "message": "limit은 1 이상 50 이하이어야 합니다."
}
```

### 필수 테스트

- 빈 문자열과 공백 문자열 거부
- `limit`의 `-1`, `0`, `1`, `50`, `51` 경계
- 지원하지 않는 `status`, `law_type` 거부
- 잘못된 조문 토큰의 일관된 오류 형태
- MCP 도구를 실제 등록 표면을 통해 호출했을 때도 같은 오류 반환

## 6. P1: 데이터 정확성 결함 수정

### 6.1 사업소개 URL 정규화

현재 사업소개 511건의 URL이 모두 `www.k-startup.go.kr/...` 형식이다.

요구사항:

- 스킴이 없으면 `https://`를 붙인다.
- 상대경로라면 공식 K-Startup 기준 URL과 결합한다.
- `http`, `https` 외 스킴은 거부한다.
- 저장 시점과 기존 스냅샷 로드 시점 중 한 곳만 정규화 책임을 가진다. 권장은 저장 경계다.
- URL이 없거나 잘못된 경우 빈 문자열로 숨기지 말고 데이터 품질 경고에 포함한다.

### 6.2 법령 범위 계산 정확일치

`_article_range_for()`는 `source_nfc in a.source` 대신 정확한 법령명 일치를 사용해야 한다. 현재 중소기업창업 지원법은 실제 67개 조문 번호인데 범위 계산은 시행령 등이 섞여 69개로 나온다.

요구사항:

- 본법, 시행령, 시행규칙을 별도 source ID로 취급한다.
- 사용자 표시명과 내부 source ID를 분리한다.
- 법령 약칭·부분일치는 검색 단계에서만 사용하고 검증·집계 단계에서는 정확 ID를 사용한다.

### 6.3 인용 귀속 신뢰도

`verify_citation()`은 조문 앞 80자 안의 최근 법령명을 찾아 귀속한다. 문장 경계를 넘어 이전 법령명에 잘못 귀속될 수 있다.

요구사항:

- 명시적 `법령명 + 제N조`를 우선 검증한다.
- 법령명이 생략된 인용은 `inferred_source` 또는 낮은 `confidence`를 반환한다.
- 문장·문단 경계를 넘은 법령명은 자동 귀속하지 않는다.
- `ok`와 `not_found` 외에 `ambiguous_source` 상태를 추가한다.

### 6.4 신선도 판정

잘못된 `fetched_at`이 현재는 경고 없이 정상처럼 처리된다.

요구사항:

- 파싱 실패는 `invalid_timestamp` 경고로 반환한다.
- 미래 시각, 두 파일의 시각 불일치, 파일 누락, `count` 불일치를 검출한다.
- 법령 데이터에도 지원사업과 동일한 신선도 정보를 제공한다.

## 7. P1: 테스트·CI·의존성 재현성

### 현재 공백

- `tests/test_server.py`는 도구 9개 등록과 안내문만 검사한다.
- 빈 성공 응답, 비정상 엔벨로프, 저장 중단, URL 정규화 테스트가 없다.
- `requirements.txt`의 `mcp`와 개발 의존성 버전이 고정되지 않았다.
- `pyproject.toml`, lock 파일, 타입 검사, 린트, CI 구성이 없다.

### 구현 요구사항

1. 프로젝트 메타데이터와 도구 설정을 `pyproject.toml`로 통합한다.
2. 재현 가능한 lock 파일을 추가한다.
3. 최소 품질 게이트를 구성한다.
   - 전체 pytest
   - 정적 타입 검사
   - 린트 및 포맷 검사
   - Python AST/컴파일 검사
4. CI는 Windows와 Linux를 모두 포함한다. 프로젝트 경로와 데이터에 한글이 있으므로 Windows 검증이 중요하다.
5. 네트워크 테스트는 실제 API에 의존하지 않고 HTTP 경계 fake를 사용한다.
6. 별도 수동 통합 테스트로 실제 API probe를 유지하되 키가 없으면 명확히 skip한다.

## 8. P2: 타입과 모듈 구조

### 문제

`law_search.py`는 707줄이며 다음 책임을 모두 가진다.

- Markdown 파싱
- 인덱스 빌드와 캐시
- 토큰화와 검색 점수 계산
- 조문 조회
- 인용 검증
- 참조 그래프 생성
- CLI

API 데이터와 MCP 반환값도 대부분 무형식 `dict`라 필드 누락이 런타임까지 드러나지 않는다.

### 권장 구조

```text
startup_law/
├── models.py          # Article, Program, 결과·오류 타입
├── law_parser.py      # Markdown → Article
├── law_index.py       # 빌드, 로드, 세대·캐시 관리
├── law_search.py      # 토큰화, 점수, 검색
├── citations.py       # 인용 검증
├── references.py      # 정·역방향 참조
├── programs.py        # 지원사업 조회
├── program_sync.py    # API 수집·정규화·원자 저장
├── law_sync.py        # 법령 수집·원자 저장
└── server.py          # MCP 경계와 도구 등록만
```

### 구현 원칙

- 구조 변경 전 현재 동작을 테스트로 고정한다.
- 기능 수정과 대규모 모듈 분리를 한 커밋에 섞지 않는다.
- `Article`은 frozen/slots dataclass로 만든다.
- 외부 JSON은 경계에서 타입이 있는 모델로 파싱한다.
- 내부 로직에서는 원시 API `dict`를 전달하지 않는다.
- 오류를 네트워크, 스키마, 데이터 품질, 파일 저장, 입력 오류로 구분한다.
- 서버 경계만 오류를 사용자용 메시지로 변환한다.

## 10. P3: 검색 품질과 성능

### 검색 품질

현재 fuzzy 옵션은 음절 bi-gram 문자 중첩이며 의미 동의어 검색이 아니다. README의 `감면 ↔ 경감` 설명은 실제 동작과 맞지 않는다.

권장 순서:

1. 문서를 현재 기능에 맞게 수정한다.
2. 법령명 약칭과 창업 전문 동의어 사전을 추가한다.
3. 대표 질문과 기대 조문으로 평가셋을 만든다.
4. BM25 또는 역색인을 도입한다.
5. 임베딩 검색은 평가셋에서 키워드 검색 대비 이득이 입증될 때만 선택적으로 도입한다.

권장 평가 지표:

- Precision@5
- Recall@10
- 첫 정답 순위(MRR)
- 인용검증 false positive/false negative 비율
- 검색 95백분위 응답 시간

### 검색 성능

현재 검색은 요청마다 전체 8,191개 조문을 순회하고 토큰별 IDF를 다시 계산한다. 로컬 MVP에서는 충분하지만 원격 동시 요청에는 비효율적이다.

권장 개선:

- 인덱스 빌드 시 정규화 텍스트와 문서 빈도 사전 계산
- 토큰 → 조문 ID 역색인
- source/law_type 필터용 별도 맵
- 인덱스 세대별 불변 캐시
- 동일 질의의 제한적 LRU 캐시
- 성능 변경 전후 벤치마크 기록

## 12. 완료 정의

다음 조건을 모두 만족해야 보완 작업이 완료된 것으로 본다.

- 빈 응답, 스키마 변경, 저장 실패가 기존 정상 데이터를 손상시키지 않는다.
- 법령 파일, 매니페스트, 인덱스가 동일한 `snapshot_id`를 가진다.
- 오래된 고아 법령 파일이 검색 결과에 포함되지 않는다.
- 모든 MCP 입력 경계가 명시적으로 검증된다.
- 사업소개 URL이 유효한 HTTPS URL로 반환된다.
- 본법 조문 범위에 시행령·시행규칙이 섞이지 않는다.
- 데이터 시각·건수·파일 불일치가 경고로 노출된다.
- 전체 자동 테스트, 타입 검사, 린트, Windows/Linux CI가 통과한다.
- CLI의 `--help`, 검색, 조회, 검증, 참조 기능을 실제로 실행해 확인한다.
- MCP 클라이언트 또는 MCP 검사 도구로 9개 기존 도구와 신규 상태 도구를 실제 호출한다.
- README의 설치법, 데이터 구조, fuzzy 검색 설명, 동기화 안전 정책이 구현과 일치한다.
