# startup-law-mcp 설계 문서

- 작성일: 2026-07-11
- 상태: 사용자 승인 완료
- 레퍼런스: [koica-reg-mcp](https://github.com/amnotyoung/koica-reg-mcp) — 구조·알고리즘·문서 스타일의 원형

## 배경과 목표

창업자가 실무에서 부딪히는 한국 법령(중소기업창업 지원법, 벤처기업법, 상법 회사편 등)을
조문 단위로 검색·조회·검증하는 MCP 서버를 만든다. 기존 법령 MCP(korean-law-mcp 등)는
전체 법령을 얕게 다루는 범용 검색인 반면, 이 프로젝트는 **창업 도메인 큐레이션 +
창업자 워크플로 특화 도구**로 차별화한다. 제작자 포트폴리오 용도를 겸한다.

**성공 기준**
- Claude Desktop 등 MCP 클라이언트에서 자연어 질의("창업 세액감면 요건 알려줘")로
  정확한 조문과 출처(citation)가 반환된다.
- LLM이 인용한 "○○법 제N조"의 실재성을 자동 검증할 수 있다.
- 법령 개정 시 동기화 파이프라인 1회 실행으로 현행본이 반영된다.

## 범위

### v1 범위 (이 설계의 대상)
- 핵심 창업 법령 15~30개 법률 + 시행령·시행규칙 (문서 약 40~60개) 큐레이션
- 조문 단위 인덱싱·검색 엔진 (순수 표준 라이브러리)
- MCP stdio 서버, 기본 5개 도구 + 창업 특화 3개 도구
- 국가법령정보센터 Open API 동기화 파이프라인
- CLI (검색 엔진 직접 실행)

### v1 제외 (후속 버전)
- 원격 배포(Fly.io streamable-http) — v2.0
- 지원사업 공고(K-Startup, TIPS) 연동 — v2.1+
- 업종 특화(음식점·이커머스 인허가) — v2.1+
- 임베딩/시맨틱 검색 — 채택하지 않음 (의존성 최소화 원칙)

### 큐레이션 법령 후보 풀

최종 목록은 구현 첫 단계에서 Open API로 법령ID·현행 여부를 조회하며 확정한다.

| 분류 | 법령 |
|---|---|
| 창업·벤처 코어 | 중소기업창업 지원법, 벤처기업육성에 관한 특별조치법, 벤처투자 촉진에 관한 법률, 1인 창조기업 육성에 관한 법률, 중소기업기본법, 소상공인 보호 및 지원에 관한 법률 |
| 설립·운영 | 상법(회사편), 부가가치세법, 조세특례제한법 |
| 고용 | 근로기준법, 고용보험법 등 4대보험 관련 |
| 지재·데이터 | 특허법, 상표법, 부정경쟁방지 및 영업비밀보호에 관한 법률, 개인정보 보호법 |
| 온라인 사업 | 전자상거래 등에서의 소비자보호에 관한 법률, 약관의 규제에 관한 법률 |

각 법률의 시행령·시행규칙을 함께 수집한다. 상법·조세특례제한법처럼 방대한 법령은
전체를 수집하되, 창업 특화 도구(단계 가이드)에서 관련 조문만 큐레이션한다.

## 아키텍처

koica-reg-mcp의 4-파일 구조를 그대로 따른다. 데이터 소스만 ALIO/HWP에서
국가법령정보센터 Open API/JSON으로 바뀌어 파이프라인이 더 단순하다.

```
startup-law-mcp/            (리포 루트 = 현재 "창업 MCP" 폴더)
├── data/
│   ├── laws/               # 법령별 .md (법종_법령명.md), Format A 정규화
│   ├── laws.json           # 큐레이션 목록 — 동기화 입력 (법령명·법령ID·분류)
│   ├── sources.json        # 매니페스트 — 동기화 산출 (법령명·법종·공포일·시행일·소관부처)
│   ├── stages.json         # 창업 단계 → 조문 매핑 (수작업 큐레이션 자산)
│   └── index.json          # 빌드 산출물 (gitignored)
├── law_sync.py             # Open API 동기화 파이프라인
├── law_search.py           # 인덱싱·검색 엔진 + CLI
├── server.py               # MCP stdio 서버 (register_tools 공유 구조 — 원격 대비)
├── requirements.txt        # mcp 하나
└── README.md               # koica 스타일: 문제→해결→도구표→로드맵
```

### 데이터 파이프라인 (`law_sync.py`)

```
laws.json (큐레이션 목록)
  → 국가법령정보센터 Open API 조회 (법령 본문 JSON — 조문 구조 포함)
  → Format A 마크다운 정규화: "### 제N조(제목)", 항·호 들여쓰기 유지
  → data/laws/*.md + sources.json 갱신
  → law_search.py build 로 인덱스 재빌드
```

- API: open.law.go.kr — 무료, OC 키(이메일 기반) 필요. **키 발급은 사용자 사전 준비물.**
- 부칙은 본칙과 네임스페이스 분리 태깅 (koica v2.2 교훈: 조번호 충돌 방지).
- "제N조의2" 가지번호, 삭제 조문(`<삭제>`) 처리.
- 현행성: v1은 수동 실행(`python law_sync.py`), v1.2에서 GitHub Actions 주간
  동기화 → 변경 감지 시 자동 PR.

### 검색 엔진 (`law_search.py`)

`koica_search.py`의 알고리즘 포팅:
- 한국어 stopword 제거 + 토큰 IDF 가중 substring 매칭
- 음절 bi-gram fuzzy (옵션) — "감면" ↔ "경감" 류 부분 매칭
- 빌드 산출물 `data/index.json`, 서버 시작 시 전 조문 메모리 로드 후 캐시
- 의존성: 표준 라이브러리만
- CLI: `search` / `get` / `verify` / `refs` / `build` 서브커맨드

### MCP 서버 (`server.py`)

- FastMCP 기반 stdio 엔트리포인트.
- `register_tools(mcp, include_admin=...)` 패턴으로 도구 정의를 한곳에 모아
  v2 원격 배포 시 HTTP 엔트리포인트가 공유하도록 설계.
- `SERVER_INSTRUCTIONS`(initialize 시 클라이언트에 전달)에 "어떤 질문에 어떤
  도구를 쓸지"를 명시 — koica가 강조한 핵심 디테일.
- 답변에 조문 출처(citation) 동봉을 지시.

## MCP 도구 세트

### 기본 5개 (koica 직접 포팅)

| 도구 | 입력 | 반환 |
|---|---|---|
| `search_law` | `query, law_type?, source?, limit?, fuzzy?` | 조문 단위 검색 결과 (snippet + citation + score) |
| `get_article` | `source, article` | 조문 본문 전체 (시행일 포함) |
| `list_laws` | `law_type?` | 인덱싱 법령 목록 + 공포일·시행일·조문 수 |
| `verify_citation` | `text` | 텍스트 내 "○○법 제N조" 인용을 ok / not_found / unknown_source / content_mismatch(제목 환각)로 분류 |
| `find_references` | `source, article, include_mermaid?` | 정방향·역방향 인용 그래프 (법↔시행령↔타법) |

### 창업 특화 3개 (차별화 — v1.1)

| 도구 | 설명 |
|---|---|
| `delegation_map` | "대통령령으로 정한다"·"○○부령으로 정한다" 위임 문구를 파싱해 법률 조문 → 시행령·시행규칙 조문을 자동 연결. 법률·하위법령 개정일 대조로 정비 지연 탐지 포함 (koica `compliance_radar`의 법령판) |
| `startup_stage_guide` | 창업 단계(아이디어 검증 → 법인설립 → 자금조달 → 첫 고용 → 세무 → 지재권)별 관련 법령·조문 큐레이션 맵. `data/stages.json` 수작업 큐레이션이 데이터 자산 |
| `check_effective_date` | 조문의 시행일·부칙 경과규정 확인 — "이 조항 지금 시행 중인가, 개정 예정인가" |

## 오류 처리

- Open API 실패(키 오류·네트워크·법령ID 변경): 법령 단위로 실패를 격리하고
  요약 리포트 반환 — 한 법령 실패가 전체 동기화를 중단시키지 않는다.
- 존재하지 않는 조문 조회: 빈 배열 + 해당 법령의 실제 조문 범위 안내
  (koica `verify_citation` not_found 패턴).
- 인덱스 부재 시 서버 기동: 명확한 빌드 안내 메시지와 함께 실패.

## 테스트 전략

- **파서/인덱서**: pytest 회귀 테스트 — 조문 수 고정값 검증, 경계 케이스
  (제N조의2, 삭제 조문, 부칙 분리, 장·절 헤더 스킵).
- **검색**: 대표 질의 → 기대 조문 top-N 포함 여부.
- **verify_citation**: 실재/비실재/제목불일치 인용 fixture.
- **E2E**: 완성 후 Claude Desktop 연결, 자연어 시나리오 5개
  (예: "창업 세액감면 요건", "법인 설립 등기 절차 근거 조문", "벤처기업 확인
  요건", "이 보고서 인용 검증해줘", "근로계약서 관련 조문") 수동 확인.
