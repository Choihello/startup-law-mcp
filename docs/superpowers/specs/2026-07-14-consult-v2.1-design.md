- 선행: v2.0 원격 배포 (startup-law-mcp.fly.dev, 로컬 13도구·원격 12도구, CI 그린)

## 배경과 목표

가장 반복되는 상담 질문은 "내가(이 사람이) 뭐 신청할 수 있나"다. 페인포인트는
놓침(커버리지)과 해석(자격 판단) 둘 다이며, 사용 시점은 **상담 중 실시간** —
속도·간결함이 우선이다. 이를 두 층으로 나눈다:

1. **서버 도구 `match_programs`** — 찾기·스크리닝. 프로필(나이·지역·예비/업력·업종)을
   공고의 자격 신호와 대조해 탈락 사유 없는 공고만 마감순으로 돌려준다.
2. **Claude Code 스킬 `startup-consult`** — 해석·상담 양식. 대화에서 프로필을
   추출하고, 결과를 상담 카드로 정리하고, 애매 조건은 법령·공고 원문 근거로
   해석하되 결론을 단정하지 않는다.

**대원칙: 판정기가 아니라 스크리닝.** 잘못된 확신이 상담에서 가장 위험하다.
파싱 불가 조건은 `unknown`으로 정직하게 표시하고 원문 확인을 안내한다 —
기존 실재성 게이트 철학의 연장.

**성공 기준**
- "만 32세, 대구, 예비창업자인데 뭐 신청할 수 있어?" → 나이·지역·예비 조건을
  전부 통과한 공고만 마감순으로, 각 조건의 근거 문구와 함께 반환
- "업력 5년 기업" 프로필에 `예비창업자` 전용 공고가 섞여 나오지 않음
- 공고 지원대상에 "여성" 등 구조화 필드로 판정 불가한 특수조건이 있으면
  **확인 필요**로 표시되어 상담 카드에 노출 (침묵 통과 금지)
- 스킬: 프로필 누락 항목을 한 번에 몰아 질문 → 카드 출력 → "업력 기산일이
  언제부터냐" 같은 해석 질문에 조문 인용으로 근거 제시 (단정 금지)
- 기존 테스트 전부 그린 유지, 신규 로직은 픽스처 기반 pytest로 커버

## 실데이터 근거 (2026-07-14 확인, 공고 289건 전수)

구상 문서는 "지원대상 자유텍스트에서 정규식 사전으로 조건 추출"을 계획했으나,
스냅샷 전수 확인 결과 **K-Startup API가 이미 구조화 자격 필드를 제공**한다:

- `target_age`: 나이 밴드의 콤마 결합 — `만 20세 미만`, `만 20세 이상 ~ 만 39세 이하`,
  `만 40세 이상` 3종 조합. 289건 전수 존재(결측 0). 3밴드 전부 = 사실상 전연령.
- `years`: 업력 밴드의 콤마 결합 — `예비창업자`, `1년미만`~`10년미만`. 결측 0.
- `region`: 시·도 단위 (`전국` 152건, `경기` 37, `서울` 34, …). 결측 0.
- 자유텍스트(`target`)의 나이 조건과 `target_age`의 **충돌 0건** — 구조화 필드를
  1차 판정 신호로 신뢰 가능.
- 단, 자유텍스트에만 존재하는 특수조건이 있음 — "여성" 언급 11건, 그 외
  재직자·대학생·소재지 상세·컨소시엄 불가 등. 이건 판정하지 않고 **감지·표시**만.

→ 설계 수정: 정규식 텍스트 추출(오파싱 위험)이 아니라 **구조화 필드 1차 판정 +
자유텍스트 특수조건 감지**의 2층 구조. 구상의 "정직한 unknown" 원칙은 유지.

## 확정 결정

| 결정 | 선택 | 근거 |
|---|---|---|
| 판정 신호 | 구조화 필드(`target_age`/`years`/`region`) 1차, 자유텍스트는 감지만 | 전수 결측 0·충돌 0 실측 — 정규식 추출보다 신뢰 가능 |
| 스크리닝 대상 | announcements의 open·closing_soon·upcoming만 | "신청할 수 있나"가 목적 — 마감·사업소개(intros)는 신청 대상 아님 |
| mismatch 처리 | 결과에서 제외, `excluded` 건수만 요약 | 상담 중 실시간 — 목록 간결성. 탈락 상세는 범위 밖 |
| keyword(업종) | 탈락 사유 아님 — `keyword_hit` 라벨만 | 업종 제한은 자유텍스트뿐이라 필터링하면 일반 공고를 오탈락시킴 |
| 정렬 | 마감순(d_day 오름차순) 고정 | 상담 실시간성 — 마감 임박이 항상 우선 |
| 특수조건 | 고정 사전 패턴 감지 → `needs_review` + 근거 발췌 | 판정기 아님 원칙 — 상담자가 원문으로 확인 |
| 도구 위치 | `programs.py` (search 관행 재사용) | 상태 계산·경고 전파·_result_row 재사용 |
| 스킬 위치 | repo `skills/startup-consult/` 공개 + `~/.claude/skills/` 설치 | 도메인 노하우의 프롬프트 자산화 — 포트폴리오 스토리 |
| 최소 입력 | 프로필 인자 최소 1개 (서버 경계에서 invalid_input) | 무인자면 list_open_programs와 중복 — 도구 목적 분리 |

## 도구 설계 — `match_programs`

위치: `programs.py` + `server.py` 등록 (로컬 13→14, 원격 12→13 도구).

```python
def match_programs(age: Optional[int] = None, region: Optional[str] = None,
                   pre_startup: Optional[bool] = None, years: Optional[float] = None,
                   keyword: Optional[str] = None, limit: int = 20,
                   today: Optional[date] = None) -> dict
```

### 조건별 판정 (제공된 프로필 인자만 검사)

- **age ↔ `target_age`**: 콤마 분리 → 밴드 파싱(`만 20세 미만` → [0,20),
  `만 20세 이상 ~ 만 39세 이하` → [20,39], `만 40세 이상` → [40,∞); 일반형
  `만 N세 이하/이상/미만`도 수용). age가 어느 밴드에든 속하면 `match`,
  전 밴드 커버(전연령)면 `match`(근거 "전연령"), 어디에도 안 속하면 mismatch(제외),
  파싱 불가 토큰뿐이면 `unknown`.
- **pre_startup=True ↔ `years`**: `예비창업자` 토큰 있으면 `match`, 없으면 mismatch.
- **years(업력 N년) ↔ `years`**: `M년미만` 토큰의 최대 M 기준 — N < max(M)이면
  `match`, 아니면 mismatch. `예비창업자` 토큰만 있으면 기창업 프로필은 mismatch.
  파싱 불가면 `unknown`. (pre_startup과 years 동시 지정은 invalid_input)
- **region ↔ `region`**: `전국` → `match`; NFC 부분일치(양방향 — "대구광역시" ↔
  "대구") → `match`; 불일치 → mismatch; 빈 값 → `unknown`.
- **keyword**: `_searchable_text` 토큰 매칭 → `keyword_hit: true/false` 라벨.
  탈락시키지 않음, 정렬도 바꾸지 않음.

각 verdict에는 근거(`evidence`) — 해당 구조화 필드의 원문 값 — 를 동봉한다.
`unknown`은 "원문 확인 필요" 의미로 상담 카드에서 확인 필요 조건으로 승격된다.

### 특수조건 감지 (`needs_review`)

`target` 자유텍스트에 고정 사전 패턴 — 여성, 대학(원)생, 재직자, 소상공인,
재도전·폐업, 컨소시엄, 소재·이전, 미가맹 등 — 이 있으면 해당 문구를
±스니펫으로 발췌해 배열로 동봉. 판정에는 쓰지 않는다.

### 출력

```json
{"results": [{
   "name": "...", "org": "...", "apply_end": "...", "d_day": 12,
   "status": "open", "url": "...",
   "checks": {"age": {"verdict": "match", "evidence": "만 20세 이상 ~ 만 39세 이하"},
              "region": {"verdict": "match", "evidence": "전국"}},
   "needs_review": ["여성 예비창업자 및 …"],
   "keyword_hit": true
 }],
 "excluded": 187,
 "warning": null}
```

- 대상: announcements 중 status ∈ {open, closing_soon, upcoming}. 마감순 정렬.
- `limit` 1~50 (기본 20), `today` 주입 가능(테스트 고정), staleness warning 전파 —
  전부 programs.py 기존 관행.
- 서버 경계: age 0~120 정수, years 0~50, limit 범위, 프로필 인자(age·region·
  pre_startup·years·keyword) 최소 1개 — 아니면 invalid_input (v1.2 규약).
- SERVER_INSTRUCTIONS에 추가: "프로필 기반 '뭐 신청할 수 있나' → match_programs
  (search_program은 키워드 검색, match_programs는 자격 스크리닝)".

## 스킬 설계 — `startup-consult`

위치: repo `skills/startup-consult/SKILL.md` + `references/eligibility-notes.md`.
설치: `~/.claude/skills/startup-consult/`로 복사 (README에 설치 안내).

### 흐름

1. **프로필 추출**: 대화에서 나이·지역·예비/업력·업종을 추출. 빠진 항목은
   **한 번에 몰아 질문** (이미 말한 건 재질문 금지). 전부 없어도 최소 1개는
   확보 후 진행.
2. **스크리닝**: `match_programs` 호출. warning이 오면 카드 상단에 표시하고
   sync_programs 안내.
3. **상담 카드 출력**: 공고별로 —
   `사업명 · D-day · 충족 조건(근거 문구) · 확인 필요 조건(unknown + needs_review) · 원문 링크`.
   마감 임박(closing_soon) 강조. 카드 하단에 "신청 전 K-Startup 원문 공고 확인"
   고지 고정.
4. **자격 해석**: 애매 조건(업력 기산일, 공동대표, 폐업 이력, 신산업 분야 등)
   질문에는 `search_law`·`get_article`·공고 원문으로 **근거를 제시**하되
   "됩니다/안 됩니다" 단정 금지 — 근거 조문 + 주관기관 확인 안내로 마무리.
5. **노하우 적립**: 상담 중 확인된 해석은 "노트에 추가해줘" 한 마디로
   `references/eligibility-notes.md`에 항목 추가 (날짜·근거 조문·요지 형식).

### eligibility-notes.md 시드

구현 단계에서 실제 인덱스를 대조하며 2~3항목 작성 (예: 업력 기산일 —
중소기업창업 지원법의 창업 정의 조문 근거; 예비창업자 정의; 신산업 분야
업력 10년 특례). **조문 인용은 verify_citation으로 실재 확인 후 기재.**
