## 1. 사실 주장 검증 결과

리포트의 검증 가능한 주장을 전부 재현했다. **전 항목 사실로 확인.**

| # | 리포트 주장 | 재현 결과 | 판정 |
|---|---|---|---|
| 1 | 빈 API 응답도 성공으로 간주, 스냅샷 0건 덮어쓰기 가능 | 코드 확인: `fetch_current_announcements`·`fetch_all`은 빈 `data`에 예외 없이 `[]` 반환 → `sync()`가 무조건 저장. 유효 JSON `{"data": [], ...}`(점검·장애 시)이면 310건 스냅샷이 0건으로 교체됨 | ✅ 사실 |
| 2 | `get_program("")`이 임의 첫 5건 반환 | 재현: 5건 반환 | ✅ 사실 (v1.1 최종리뷰 M4로 이미 인지, defer였음) |
| 3 | `limit=-1`이 대량 결과 반환 | 재현: `search("창업", limit=-1)` → **425건** (`scored[:-1]` 음수 슬라이스) | ✅ 사실 (신규 발견) |
| 4 | 사업소개 URL 전건 스킴 없음 | 재현: intro **511/511건** `www.` 시작, 공고는 0/310 정상 | ✅ 사실 (신규 발견) |
| 5 | `_article_range_for`가 본법·하위법령 혼입 (67 vs 69) | 재현: range 표기 "제1조 ~ 제66조, 총 **69**개" vs 정확일치 본법 **67**개 — 리포트 수치와 일치 | ✅ 사실 (v1.0 리뷰에서 Minor 인지) |
| 6 | 잘못된 `fetched_at`이 무경고 통과 | 재현: `staleness_warning({"fetched_at": "garbage"})` → `None` | ✅ 사실 |
| 7 | `build_index`가 매니페스트가 아닌 `*.md` 전체 glob | 코드 확인: law_search.py:141 `LAWS_DIR.glob("*.md")` — 큐레이션 제거·이름 변경 시 고아 파일 잔존 | ✅ 사실 |
| 8 | README fuzzy 설명("감면↔경감")이 실동작과 불일치 | README 248행 확인. "감면"과 "경감"은 공유 bi-gram이 없어 **이 예시는 매칭 불가능** — 문서 오류 맞음 | ✅ 사실 (신규 발견) |
| 9 | verify_citation 80자 창이 문장 경계 넘어 오귀속 가능 | 코드 확인: `nearest_source`가 prefix 80자 `rfind` — 설계상 사실 | ✅ 사실 (알려진 한계) |
| 10 | 두 스냅샷 파일 쓰기 사이 실패 시 세대 불일치 가능 | 코드 확인: `write_text` 2회 직접 호출 | ✅ 사실 (v1.1 최종리뷰 M2 — 더 오래된 쪽 기준 신선도로 완화됨을 리포트는 미언급) |
| 11 | `count` vs 실제 `items` 길이 불일치 미검출 | 코드 확인: `load_programs`는 `count` 무시 | ✅ 사실 |
| 12 | 검색이 매 요청 전 조문 순회 + IDF 재계산 | 코드 확인: 사실. 로컬 성능은 실측 1초 미만으로 문제없음 | ✅ 사실 (로컬에선 비문제) |

**평가**: 이 리포트는 실제 코드를 읽고 쓴 것이며, 수치(67/69, 511건)까지 정확하다.
특히 #1(빈 응답 덮어쓰기), #3(음수 limit), #4(URL 스킴), #8(README 오류)은 우리
내부 리뷰 게이트가 놓친 **신규 발견**으로, 리포트의 실질 기여다.

## 2. 우선순위별 타당성 평가

### P0-1 지원사업 스냅샷 보호 — **타당, 채택 권고 (일부 완화)**

핵심 위험이 실재한다: `sync_programs`는 MCP 도구라 사용자가 자연어로 수시 실행할 수
있고, 법령과 달리 **git 검토 없이 파일이 바로 교체**된다. 빈 응답 한 번이면 310건이
0건이 된다. 이건 "기능 추가보다 정확성 우선" 원칙에 정확히 부합하는 P0가 맞다.

다만 세부 요구 중 조정 권고:

- ✅ 채택: 0건 방어, 급감 방어(정책값 상수화), 응답 스키마 검증, 임시파일→교체,
  before/after 건수 반환
- ⚠️ 완화: `snapshot_id` 교차검증 + 조회 시 이상 경고(요구 5~6번)는 두 파일에 동일
  `fetched_at`을 쓰는 현 구조에서 `fetched_at` 비교로 충분히 대체 가능. 별도 ID
  체계는 이득 대비 스키마 변경 비용이 큼 → `fetched_at` 일치 검사로 축소 권고
- 급감 기준 70%는 합리적 초기값 (공고 310건이 주 단위로 93건 이하로 줄어드는 건
  비정상이 맞음)

### P0-2 법령 동기화·인덱스 세대 일치 — **문제는 실재하나 P0는 과대평가, 축소 채택 권고**

리포트가 놓친 맥락: 법령 동기화는 지원사업과 **운영 흐름이 다르다**. `law_sync.py sync`는
CLI 수동 실행이고 산출물이 **git 커밋으로 검토·되돌리기 가능**하다. MCP 도구로 노출돼
있지도 않다(v1.0에서 의도적으로 admin 도구를 만들지 않음). 따라서 "구법·신법 혼재"는
데이터 유실이 아니라 **재현 가능한 운영 실수** 범주로, P1이 적정하다.

- ✅ 채택 (저비용·고가치): 매니페스트 기반 인덱싱(요구 2 — glob 제거, 몇 줄 수정),
  큐레이션 제거 법령의 고아 파일 정리(요구 6), build 후 캐시 무효화(요구 8 —
  v1.0 최종리뷰 때부터 v2 전 해소 예정이던 항목)
- ⚠️ 보류: staging 디렉터리 + 파일별 sha256 + `index_built_at` + 세대 ID 검증(요구
  1·5·7)은 주간 수동 sync + git 검토 흐름에서 이득이 작음. v1.3(GitHub Actions
  자동화)에서 sync가 무인화될 때 도입하는 게 시점상 맞음

### P1 MCP 입력 검증 — **타당, 전면 채택 권고**

세 재현 사례 모두 확인됐고, 수정 비용이 낮다(경계 함수 하나 + 각 도구 앞단 검증).
`limit=-1` → 425건은 LLM이 실수로 넣을 수 있는 값이라 실사용 위험도 있다. 구조화
오류 형식(`{"status": "invalid_input", ...}`)도 기존 `sync_programs` 오류 형식과
일관된다. 이견 없음.

### P1 데이터 정확성 4건 — **3건 채택, 1건 부분 채택**

- **6.1 URL 정규화**: 채택. 511건 전건 재현. 저장 경계(normalize_intro)에서 처리하자는
  제안도 우리 구조와 맞음. 단, 이미 커밋된 스냅샷 511건은 다음 sync 전까지 그대로이므로
  **로드 시점 1회 보정 또는 재동기화 1회 병행** 필요 — 리포트는 이 마이그레이션을 명시하지 않음.
- **6.2 범위 계산 정확일치**: 채택. 단 심각도 인지 필요 — 이 값은 `verify_citation`의
  not_found **안내 메시지에만** 쓰이고 검증 판정 자체는 정확일치라 실판정 오류는 없다.
  "본법·시행령 source ID 분리" 같은 스키마 변경 없이 `in` → `==` 한 줄이면 된다.
  리포트의 "표시명/내부 ID 분리" 제안은 과설계.
- **6.3 인용 귀속 confidence**: 부분 채택. `ambiguous_source`/`inferred_source` 상태
  추가는 가치 있음(환각 검증 도구의 정직성 향상). 반면 "문장·문단 경계 파싱"은 한국어
  문장 경계 검출 자체가 새 오류원이라, 80자 창 + confidence 표시가 비용 대비 나음.
- **6.4 신선도 판정**: 채택. `invalid_timestamp` 경고, 미래 시각, count 불일치 검출
  모두 저비용. "법령에도 동일 신선도 제공"은 data_status(P2)와 묶는 게 자연스러움.

### P1 테스트·CI·의존성 — **부분 채택 권고**

- ✅ 채택: pytest GitHub Actions CI (Windows+Linux — 한글 경로·cp949 이슈를 이미
  겪어서 Windows 검증 필요성 주장은 설득력 있음). 리포트의 신규 테스트 10개 목록은
  P0·P1 수정의 회귀 테스트와 대부분 겹치므로 자연 흡수됨.
- ⚠️ 보류: `pyproject.toml` 전환 + lock 파일 + mypy + 린트 일괄 도입은 "의존성이
  mcp 하나"인 프로젝트에서 이득이 작다. 포폴 어필로는 CI 배지 하나가 lock 파일보다
  효과적. 원하면 v1.3에서 Actions와 함께.

### P2 타입·모듈 분리 — **보류 강권**

여기가 리포트와 가장 크게 갈리는 지점이다. 9개 모듈 + 패키지 구조로의 분리는:

1. **의도된 설계와 충돌**: 이 프로젝트는 koica-reg-mcp(단일 파일 1,438줄)의 "파일 몇
   개로 전부"라는 미학을 계승했고, 이는 README 기술 메모에 명시된 차별점이다.
   707줄은 레퍼런스의 절반이고 아직 책임 경계가 코드 순서로 잘 드러난다.
2. **위험 대비 이득 부족**: 대규모 이동은 그 자체로 회귀 위험이고, 얻는 건 지금
   시점엔 import 경로 정리뿐이다. 리포트 스스로도 "구조 변경 전 동작을 테스트로
   고정하라"고 쓸 만큼 위험을 인지하고 있다.
3. **시점 문제**: 분리가 정당해지는 건 v2 원격(HTTP 엔트리포인트 추가)이나 v1.2 특화
   3도구로 코드가 실제로 더 붙을 때다. 지금은 YAGNI.

frozen dataclass·TypedDict 정도의 가벼운 타입 강화는 P0·P1 수정에 얹어가는 수준으로만.

### P2 data_status 도구 — **채택 권고**

저비용(두 스냅샷 메타 읽어 합치는 읽기 전용 도구)에 운영·포폴 가치가 모두 있고,
6.4 신선도 요구의 자연스러운 수용처다. 도구 수 9→10.

### P3 검색 고도화 — **순서만 채택**

리포트의 권장 순서(문서 수정 → 동의어 사전 → 평가셋 → BM25 → 임베딩은 입증 시에만)는
정확히 올바른 순서다. 다만 지금 실행할 것은 **1번(README 감면↔경감 수정)뿐**이고
이건 P3가 아니라 5분짜리 문서 수정이다. 나머지는 v2 원격에서 성능 요구가 실측될 때.

## 3. 리포트가 놓친 것

- **완화 장치 미언급**: #10(두 파일 세대 불일치)은 `load_programs`가 더 오래된 쪽
  `fetched_at`으로 신선도를 판정해 이미 보수적으로 완화돼 있다. #1의 XML 오류
  엔벨로프는 `_get_json`이 이미 RuntimeError로 잡는다(v1.1 Task 1 fix). 리포트는
  현 방어선을 0으로 놓고 서술해 위험을 다소 부풀린다.
- **운영 흐름 차이 미반영**: 법령(수동 CLI+git 검토) vs 지원사업(MCP 도구로 즉시
  교체)의 위험도 차이를 같은 P0로 묶었다 (→ §2 P0-2).
- **URL 기존 데이터 마이그레이션**: 저장 경계 정규화만으로는 커밋된 511건이 안 고쳐진다.
- **기존 백로그와의 중복**: 캐시 무효화, get_program 가드, d_day DRY 등은 v1.0·v1.1
  최종리뷰 defer 목록과 겹친다 — 새 발견처럼 서술됐지만 이미 추적 중이던 항목.

## 4. 리포트의 실행 지시문(§14)에 대한 의견

"P0·P1만 구현, 실패 테스트 먼저, 데이터 무손상, 기능 수정과 구조 분리 분리" — 지시문
자체는 우리 작업 방식(TDD·태스크별 리뷰 게이트)과 정확히 일치하며 그대로 써도 된다.
단 P0-2는 위 §2대로 축소한 범위로 해석할 것을 전제한다.

## 6. 결론
